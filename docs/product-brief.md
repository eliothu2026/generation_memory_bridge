# 产品简报 · 代际记忆桥梁（Generation Memory Bridge）

> 本文是这个项目的**产品文档**：它回答"给谁做、解决什么问题、怎么算成功、
> 有哪些风险"，与偏技术的 [README](../README.md) 互补。
>
> **诚实声明**：这是一个处于原型阶段的个人项目。下文中的用户画像与场景是
> **待验证的假设**（标注为 🔵 假设），不是已完成的用户研究（标注为 ✅ 已验证）。

## 1. 一句话定位

**祖孙对话的实时副驾：在年轻人听长辈口述时实时补位（理清逻辑、补全背景、看见物件、
读出情绪），让这场对话更听得懂、接得住、更有温度。**

> 关键区隔：我们的产品是**那场对话本身（的体验）**，不是事后那本回忆录。结构化的
> 记忆档案是**副产品**——这与几乎所有竞品（以"成书"为最终交付）根本不同。

---

## 2. 问题与机会

### 问题
长辈的人生记忆珍贵且易逝；但真正卡住"祖孙对话"的，是**听的那一方（年轻人）当场就
跟不上、接不住**：

- **老人讲得跳**：想到哪说到哪，没有时间线、缺背景，年轻人听着像一团乱麻。
- **年轻人缺语境**：老人随口提的"记工分/粮票/下乡"，年轻人没概念，**当场无法理解，
  也就无法共情、无法追问**——对话很快就聊不下去、变尴尬。
- **不知道怎么接**：老人讲到动情处，年轻人常因为"不知道说什么"而错过了最该回应的时刻。
- **窗口正在关闭**：能讲的人越来越少，这是一个**不可逆、有时间压力**的需求。

> 现有工具把力气全花在"事后把录音变成一本书"，却没人解决**对话当下**的体验。

### 为什么是现在（Why now）
三项 AI 能力刚好同时成熟，使"边听边理解"第一次变得可行且便宜：

- **LLM 推理**便宜到可以对每段口述做结构化整理与背景推断；
- **ASR（语音识别）**足够准，能实时把方言/口语转成文本；
- **文生图**能把"记忆里的物件"低成本地可视化。

三年前做这个，成本和效果都不成立；现在成立了。这是一个**能力拐点驱动的机会**。

## 3. 目标用户

### 主要用户 🔵（假设，待验证）
**想听家中祖辈口述回忆的年轻人（约 18–35 岁孙辈/子女）。** 他们有意愿亲近长辈、
有设备、会用 AI，但**缺历史背景、不擅长引导对话**——一坐下来就冷场，老人讲的很多
内容他们听不懂、接不住。产品是给**听的这一方**的实时辅助。

> 典型心声："我想多听我爷爷讲讲以前，但他讲得我经常听不懂、也不知道该问什么、
> 该怎么接，聊几句就冷场了。"

### 次要用户 🔵
- **讲述者本人（65+ 老人）**：真正的内容来源，但通常不是付费/操作的人。产品对他们的
  要求必须**极低门槛**——最好只需要"说话"。
- **机构用户**：养老机构、地方志/方志办、口述史研究者、博物馆。需求更结构化，是
  潜在的 B 端延展，但**不是 MVP 重点**。

### 谁不是我们的用户（明确排除）
- 想要专业级影视访谈制作的人（那是高成本人工服务）。
- 把它当通用录音/笔记工具的人（我们不做通用，只做"记忆整理"这一件事）。

## 4. 核心用户场景（Jobs-to-be-done）

核心 JTBD 是**提升祖孙对话当下的体验**——让年轻人"听得懂、接得住、有共鸣"：

1. **实时理清**：老人讲得跳，AI 在旁把零散讲述实时整理成时间线，年轻人随时看得清脉络。
2. **背景补位**：老人随口提的时代名词（记工分/粮票/下乡），AI 实时给年轻人一句通俗解释
   并核实，年轻人当场就懂，能接话、能追问。
3. **看见记忆**：把老人反复提起的"老物件/老地方"生成画面，作为情感锚点，激发更多回忆、
   也让对话更生动。
4. **读出情绪、提示回应**（规划中的第 4 条流水线）：感知老人叙述中的情感波动，在恰当
   时机**轻轻提示年轻人**——该停下来共情，还是顺势追问一个细节，避免错过最该回应的时刻。
5. **副产品：留存**：以上过程**顺带**沉淀出一份可读的记忆档案，可分享/传承——但这是
   副产品，不是产品的核心价值所在。

## 5. 竞品与差异化

> 调研发现：海外（StoryWorth、Remento、Autobiographer、Life Story AI、Storii…）与
> 国内（柠檬优力回忆录工坊、各类 AI 代写）**几乎清一色是"问答/出题 → 转写 → AI 写成
> 一本书"**，最终交付物是**书**。行业还公开承认 AI 的硬伤：方言转写难、**易凭空脑补**
> （媒体案例：老人讲贫苦童年，AI 却插入"阳光明媚"——典型幻觉），仍需人工把关。

| 产品 | 交付物 | 模式 | 缺口（我们的机会） |
|---|---|---|---|
| **StoryWorth** | 精装书 | 每周邮件出题、手写 | 纯异步、不理解、不实时 |
| **Remento** | 书 | 语音口述→AI 成文 | **事后成书**，非实时、不补背景 |
| **Autobiographer** | 数字自传 | AI 对话追问 | 重隐私，仍是"生成一份记录" |
| **Storii** | 音频/PDF | 电话录音（适老） | 适老化好，但只录不理解 |
| **柠檬优力·回忆录工坊** | 书/文档 | 主题引导+语音转写+家庭协作 | 国内代表，仍是"成书"，无实时理解/核验 |
| **传统录音/转写**（飞书妙记等） | 逐字稿 | 通用转写 | 只记录不理解 |

**我们的差异化（护城河假设）**：
0. **【核心区隔】产品是"对话体验"而非"那本书"**——所有竞品都在优化"事后成书"，
   我们优化"对话当下"；记录只是副产品。这是品类层面的不同，不是功能多寡之争。
1. **实时"边听边理解"**——讲述当下就在理清逻辑、补背景，而非事后处理；
2. **背景还原 + 实时考据**——补全时代语境并用维基核实，**直接对冲行业公认的"AI 脑补/
   幻觉"痛点**（"阳光明媚"问题）；
3. **记忆可视化**——把抽象记忆变成看得见的画面；
4. **情感感知 + 互动提示**（规划中）——读出老人情绪、提示年轻人如何回应，没有竞品在做。

> ⚠️ 这三条目前是**设计假设**，未经用户验证哪条最打动人。验证哪条是真需求，是
> 下一步最重要的产品工作（见第 9 节）。

## 6. 范围与非目标（Scope / Non-goals）

**MVP 做什么**：实时把一段口述整理成 ① 时间线大纲 ② 背景知识 ③ 记忆锚点+生图；
支持逐段累积、断点续接；产出可读档案。

**MVP 明确不做（划清边界本身是产品决策）**：
- ❌ 不做通用聊天机器人 / 通用笔记。
- ❌ 不做"复活逝者"的数字分身——刻意避开 HereAfter 式的伦理雷区。
- ❌ 不做多人实时协作、不做高并发服务化（原型阶段不需要）。
- ❌ 不追求史实考据级准确——我们整理的是**个人记忆**，不是历史教科书（这也直接影响
  下一节的风险设计）。

## 7. AI 特有的风险与对策 ⭐

> 这一节是 AI 产品与普通产品的分水岭。一个 AI PM 的核心价值，是**预判模型会怎么坑用户**
> 并设计护栏。

### 风险 1：幻觉 / 虚构记忆（最高优先级）
LLM 在"背景还原"时可能**编造**不存在的史实，甚至把编造内容混进老人的记忆里。对一个
"记忆传承"产品，**污染真实记忆是不可接受的伤害**。

- **对策**：严格区分两类内容——「**老人原话**」(只整理、不杜撰) 与「**AI 补充的背景**」
  (清晰标注"这是 AI 推测的时代背景，供参考")。在 UI 上视觉分离，让用户永远知道
  "哪些是我爸说的，哪些是 AI 猜的"。

### 风险 2：虚假的置信度
当前原型里"背景还原"会输出 `置信度 0.98`——但这个数字是 LLM 自己编的，**没有任何
校准依据**。给用户一个看似精确、实则无根据的数字，比不给更危险。

- **对策**：要么用**离散的、诚实的分级**（"较确定 / 推测 / 存疑"）替代假精确的小数；
  要么建立真正的校准机制后再展示数字。**宁可模糊但诚实，不要精确但虚假。**

### 风险 3：质量无法评估（eval）
"整理得好不好"目前全靠肉眼。没有评估，就无法迭代、无法对用户承诺质量。

- **对策**：定义可评估的质量维度（时间线是否符合原意、背景是否有据、有无杜撰），
  建小型人工标注集做回归，逐步引入自动评估。

### 风险 4：成本与延迟是产品问题，不只是技术问题
单段分析需 1–2 分钟、3 次 LLM 调用。一次完整口述的成本与等待，直接决定商业模式与体验。

- **对策**：技术上已用**合并队列（背压）**把"分析次数与上游语速解耦"（见 README）；
  产品上需把等待"翻译"成体验——例如让用户在 AI 整理时继续聊，不空等。

### 风险 5：隐私与伦理
老人的人生故事 = 极度敏感的个人数据，且讲述者（老人）与决策者（子女）常不是同一人。

- **对策**：数据本地优先/可导出可删除；明确**知情同意**（老人是否愿意被记录、被分享）；
  谨慎处理涉及在世他人的隐私内容。

### 风险 6：情感提示可能把"真情"变成"念稿"（针对规划中的情感 agent）
实时提示年轻人"该共情/该追问"，若做得过度，会让年轻人变成照屏幕念问题，**反而破坏
祖孙之间最珍贵的真诚连接**；且纯文本难准确判断情绪，误读会很尴尬。

- **对策**：① 信号上先做文本、诚实标注"漏掉声音里的情绪"，架构可插拔以后接语音情感识别；
  ② UX 必须**克制、可选、不打断**——是"卡壳时轻轻递一个建议"，不是"指挥你说话"；
  ③ 区分**共情提示**与**追问提示**，按情绪强度选择，宁可少提示也不要打断动情时刻。

## 8. 成功指标

> 北极星指标要衡量"用户真正获得的价值"，而非虚荣指标（如注册数）。

- **北极星指标（候选）**：**被完整保存/分享给家人的记忆档案数**——它同时代表
  "整理质量够好"+"用户觉得值得传出去"，直击产品的终极价值。
- **辅助指标**：
  - 激活：完成"第一段口述并看到整理结果"的用户比例（首次体验是否打动人）。
  - 深度：人均累积的口述段数 / 时间线事件数（是否愿意持续聊下去）。
  - 质量：整理结果的人工评估准确率、"无杜撰"达标率。
  - 情感信号：生图锚点的保存/分享率（可视化是否真的制造了情感价值）。

## 9. 当前状态与下一步

### 已验证 ✅
- 技术可行性：三条流水线能在流式输入下实时、并发地产出结构化结果（见 README 实测）。
- 工程主干：流式 + 背压 + 并发 + 断点续接已跑通。
- 真实输入闭环：本地 faster-whisper 把**录音转成文字**并接入分析，端到端实测通过
  （音频 → 文字片段 → 三条流水线），离线、免 key。
- 体验门槛：**免 key 回放试用**已落地，无需配置 key、零成本即可完整体验产品。
- 抗幻觉护栏：**考据 agent**（自主调用维基核实史实、保守标注/删改）已端到端实测通过
  （真实 LLM + 真实维基核实，背景笔记可见「✅据维基核实」标注）。
- 框架无关：在反复评估后**彻底移除 CrewAI**，三条变换与考据 agent 均改为直连 LLM，
  真实运行实测无退化；依赖从 ~1GB 瘦身为两个小库。
- 体验型能力首发：**建议追问（第 4 条流水线）v1** 已落地——替年轻人实时想细节追问、
  自我克制不打扰，是"体验即产品"定位的首个直接落点（demo 可见）。

### 未验证 🔵（这是诚实的缺口）
- **需求真伪**：还没有任何真实用户证明这三件事里"哪件最值钱"。
- **可用性**：老人这一端的真实门槛、子女这一端的真实意愿，都未测。
- **识别质量**：ASR 在真实方言/口音/嘈杂环境下的准确率尚未系统评估。
- **说话人分离**：当前 ASR 产出一条不分人的文字流，分不清"长辈叙述"与"年轻人提问"；
  双人对话 + 追问场景下会污染分析，需 diarization（whisperx/pyannote）或输入分通道，尚未做。

### 下一步（按学习价值排序）
1. **最小可用验证**：给 3–5 对真实祖孙试用，验证核心 thesis——"实时副驾真的提升了对话
   体验吗？年轻人更愿意听、更接得住了吗？"（这比"哪条流水线"更根本）。
2. **情感感知 + 互动提示（第 4 条流水线）**：先做文本版 v1，验证"提示年轻人如何回应"
   到底是加分还是打扰（第 7 节风险 6）——这是"体验即产品"定位最纯粹的落点。
3. **风险护栏打磨**：原话 vs AI 推测的视觉分离；考据 agent 也核年代估计本身。
4. **打磨展示**：演示动图、真实生图模型（见 README 路线图）。

## 10. 关键决策记录（Decision Log）

> 记录"在什么约束下、考虑了什么、为什么这么选"。这些既是工程决策，也是产品决策——
> 因为它们最终都服务于用户体验。

| 决策 | 约束 / 触发 | 选项权衡 | 结论 |
|---|---|---|---|
| **从"一次性分析"转向"流式运行时"** | 真实输入是无界语音流，单段分析 1–2 分钟，与亚秒级上游存在 100× 吞吐错配 | 每句都分析（被冲垮）/ 排长队（延迟爆炸）/ 合并处理 | 引入**合并队列（背压）**：上游再快只让单段合并更长，分析次数与语速解耦 |
| **砍掉 CrewAI `Flow`** | Flow 的"一次跑完一张 DAG"模型不适合无界流；既背概念成本又用不上其编排能力 | 真用 Flow / 去掉 Flow | **保留擅长干活的 `Crew`，去掉 `Flow` 这层壳**，用 async 事件循环统一输入/并发/服务 |
| **进一步彻底移除 CrewAI** | 砍掉 Flow 后再评估：三条变换退化成"填模板→调一次 LLM→解析"，用不上多 agent/工具能力；这套"独立变换+核验者"的拓扑里 CrewAI 概念最不匹配、体量最重（30 依赖、~1GB） | 留着 CrewAI（统一但重）/ 迁 PydanticAI/LangGraph（更贴合）/ 直连无框架 | **直连 DeepSeek + Pydantic，无框架**；唯一需要 agent 能力的考据器手写工具循环（~百行）。依赖从 ~1GB 瘦身为 openai+httpx，已实测无退化 |
| **状态持久化用 SQLite 起步** | 需要"崩溃可续接"，但又不想让使用者去装/维护数据库服务 | 内存（会丢）/ Redis/PG（要起服务）/ SQLite（零依赖落盘） | 选 **SQLite**：零额外依赖即可断点续接；接口按可换 Redis/PG 设计，为扩展留缝 |
| **生图暂用 stub** | 原型阶段，先验证"理解"主干而非"输入输出"两端 | 一步到位接真实模型 / 先占位 | **先占位**，把精力投在最不确定的"理解"环节，降低验证成本 |
| **真实 ASR 选本地 Whisper 而非云 API** | 项目一以贯之的"零门槛、可离线、免 key"定位；云 ASR 会把"再配一个 key、再花一笔钱"的门槛带回来 | 云 ASR（快、支持实时流，但要 key/花钱）/ 本地 faster-whisper（免费离线，CPU 偏慢） | 选 **本地 faster-whisper**：契合开源/免 key 气质；设为可选依赖，不拖累只看文本 demo 的轻量用户 |
